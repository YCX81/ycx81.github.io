🔧 嵌入式知识：LoRaWAN：区域参数、空中时间与可靠性

本节已于 2026-07-14 按官方资料重新校订；旧版自动生成内容已移除。

LoRa 是调制，LoRaWAN 是端到网络的协议

LoRaWAN 定义 MAC、帧、安全、激活、数据率和设备类别等；LoRa 只是常见物理层调制。典型 star-of-stars 中，end-device 的 uplink 可被多个 gateway 同时接收，gateway 转发给 Network Server 去重和处理。gateway 不是 Zigbee 式路由节点，设备也不应绑定某一台 gateway。

所有 LoRaWAN end-device 都要支持 Class A：设备发送 uplink 后打开 RX1、RX2 两个短接收窗口，网络下行必须配合这些机会。Class B 增加由 beacon 同步的周期 ping slot；Class C 在不发送时尽量持续接收，降低下行等待但显著增加功耗。选择类别是下行时延、功耗、同步和网络容量的系统权衡，不能把 Class C 当作“更高级的默认模式”。

每个区域的信道、数据率、最大发射功率、payload 限制和法规约束由 Regional Parameters 及当地法规决定。EU868、US915、AS923、CN470 等不能只改一个频率常量便互换。产品必须固定目标 region、信道计划和栈版本，并按对应认证配置验证。

空中时间是共享预算

Spreading Factor 增大通常改善接收能力，却显著增加 time-on-air，减少网络容量并增加设备能耗。ADR 可由网络依据链路情况调节数据率和功率，适合链路条件较稳定的设备；高速移动、信号快速变化或长期收不到下行时要按规范处理 ADR 状态，不能把一次实验室参数永久锁死。

Confirmed uplink 会请求网络确认，但 ACK 仍受下行窗口、gateway 可用性、占空比/驻留时间和网络调度限制。对每个采样都 confirmed 并无限重试会放大拥塞。关键业务应使用应用序号、批量补报、去重和有限退避；网络确认也不证明云端业务或物理动作完成。

OTAA 应使用唯一的根密钥和标识完成激活并导出 session keys；不能在整批设备中复制同一凭据。frame counter、DevNonce/JoinNonce 及规范要求的持久状态必须正确处理掉电和回滚，否则可能被 Network Server 作为重放拒绝。密钥不能进入日志或普通固件导出区，恢复出厂设置也必须与网络侧重新入网流程协调。

常见误区与适用边界

  • 用 RSSI 好看就认定链路可靠。还应观察 SNR、gateway diversity、位置/季节变化、丢包与重试。
  • 用 LoRaWAN 做持续流或频繁远程控制。它面向低速、小载荷、非实时遥测，不适合音视频和严格低时延闭环。
  • 将 gateway 收到包等同于应用收到。Network Server 去重、MIC/counter 校验、Application Server 解密和业务入库都可能失败。
  • 忽略天线与法规。错误匹配、外壳、地平面、安装方向和地区限制可让“软件参数优化”失去意义。

工程检查清单

  • 固定 TS001/RP002 版本、region、默认信道、RX1/RX2、Class 与 ADR 策略。
  • 用真实 payload 长度计算并测量 time-on-air、每日 airtime、峰值电流和预计电池寿命。
  • 为 join、counter、MAC command 和队列状态设计掉电安全持久化，并测试 flash 回滚。
  • 统计 uplink 成功率、gateway diversity、SNR/RSSI、confirmed 重试、下行失败和应用缺号。
  • 测试无覆盖、网络拒绝 join、长时无下行、gateway 离线、密钥错误、计数器异常和满队列。
  • 用目标区域的射频/LoRaWAN 认证方案验证，不以开发板互通替代合规。

可验证示例

在目标区域配置下，让设备每 10 分钟发送含递增业务序号的小 payload，运行至少 72 小时。记录每帧数据率、Tx power、time-on-air、FCnt、接收 gateway 数、SNR/RSSI 与耗电。期间屏蔽网络 2 小时并让设备掉电重启，随后恢复覆盖。验收应证明：退避有上限、队列不无限增长、计数器不回退、重复帧不重复入库、缺口可观测，并且 ADR 恢复符合所用 TS001/RP002 版本。
