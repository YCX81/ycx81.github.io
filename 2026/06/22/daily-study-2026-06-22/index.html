🔧 嵌入式知识：电源管理与低功耗设计

原理

嵌入式产品的电池寿命、散热、整机成本，几乎 80% 取决于电源管理设计。“低功耗”是一个系统级话题，涉及硬件选型、电源拓扑、外设使能、软件调度全方位。

芯片低功耗机制（以 Cortex-M 为例，具体名称因厂家不同）：

  • Run：正常运行，每 MHz 数十 μA。
  • Sleep：CPU 停，外设继续；任意中断唤醒。
  • DeepSleep / Stop：CPU 和多数外设关，SRAM/寄存器保持；RTC/EXTI 唤醒；典型 μA 级。
  • Standby：只保留 RTC 和少数备份域，SRAM 丢失；唤醒相当于复位；典型 nA~μA。
  • Shutdown / VBAT：除 VBAT 供电的 RTC 外全断；最低功耗但要软件处理“完全复位”。

硬件层面的降功耗手段：

  • 电源分区：把只偶尔用的外设放在独立 LDO 下，不用就整块关。
  • 动态调频 DVFS：CPU 和外设按需调频压缩功耗 = CV²f。
  • 电源域电压调整：高功耗模式 1.2V，低功耗 1.0V。
  • 时钟门控：不用的外设时钟关闭（STM32 RCC 寄存器）。
  • 上拉电阻与弱上拉：长期悬空的 IO 会漏电，未用脚要配成模拟态或下拉。
  • LDO vs DC/DC：低压差 LDO 简单但效率差（Vout/Vin）；DC/DC 效率高但引入纹波，二者常级联。

软件调度：

  • 中断驱动 + WFI：能睡就睡，WFI 让 CPU 进 sleep；DMA + RTOS tickless 模式是黄金组合。
  • Tickless IDLE：普通 RTOS 周期 SysTick 会频繁唤醒，tickless 模式只在有任务时才醒。
  • 关闭外设：退出使用立即关；例如 ADC 完一次测量就 disable。
  • 减少 flash 读取：指令 Flash 读耗电大，循环里放 RAM 函数可省功耗（但通常非瓶颈）。
  • 通信批量化：把零散的短消息攒一起集中发一次，发射 peak 电流很高，次数越少越省。

典型电池寿命估算：寿命 = 容量 / 平均电流，平均电流 = Σ(状态电流 × 状态占比)。做好表格：Run xx mA × yy%，Sleep xx μA × zz% …… 才能抓住真正的大头。

易混淆点

  1. 电流瞬时峰值 vs 平均：Tx 瞬间 100mA，平均算起来占 0.01% 的话仍可能是日耗电大头；要算能量 I × t。
  2. RTC 唤醒 ≠ 复位：Stop 模式唤醒后程序继续往下跑；Standby 唤醒则是从复位向量重新开始。
  3. Flash 断电写保护：低电时写入会坏页，必须先做电压监测（PVD）或 brown-out detection，关键数据写 FRAM 或双备份。
  4. 外设时钟门控后再配置？：有的外设寄存器必须先开 Clock 才能访问，关了时钟再读会 bus fault；顺序要严格。
  5. LDO 全程高压差：USB 5V 下一级直降到 1.8V 核心域，LDO 效率只有 36%；换 DC/DC 能省一大截电。
  6. 睡眠漏电：GPIO 配成输入且内部上拉一直导通的话每脚漏几 μA，几十个 IO 就是 mA 级差。

面试常见问法

Q1：怎么测 MCU 的低功耗？
A：把板子串进高精度电流表（μA 级），分别在 Run / Sleep / Stop / Standby 下测，记录稳态电流。短时峰值用示波器 + 串联 shunt 电阻（或 Joulescope）。做法是把 MCU 单独断电，确认其他外设电流不掺进来，才能反映“真 MCU 功耗”。

Q2：STOP 模式唤醒为什么变慢？
A：Stop 模式主时钟 HSI/HSE + PLL 关闭，唤醒后必须重启 PLL，等稳定（几百 μs），才能恢复高频运行。硬件唤醒中断能做到 μs 级进入 ISR，但要运行到业务代码还需时钟重配。对延迟要求紧的系统可考虑 Sleep 模式（CPU 停、时钟在）。

Q3：电池产品怎么做“无线休眠”？
A：BLE/LoRa 之类在连接或广播间隔大部分时间让射频芯片深度睡眠、MCU 进 STOP，仅在事件窗口唤醒。MCU 与 RF 之间通过 wakeup 引脚联动。软件上要 batch 收发，加少广播间隔 → 大部分时间休眠，电池可跑 1~3 年。

Q4：低功耗设计最大的坑是什么？
A：“忘了关”——LED 背光、上拉电阻、调试串口、某个外设驱动 …… 只要一处没关就拉高静态电流。工程上应有一张“休眠资产清单”：每一路电源域、每一个外设、每一个 IO 都标注睡眠策略，一项一项确认。
