🔧 嵌入式知识：链接脚本与内存布局

原理

链接脚本（Linker Script，通常为 .ld 文件）是 GNU ld 链接器的配置文件，用于告诉链接器如何把多个目标文件（.o）中的段（section）放置到最终可执行映像的内存地址空间。嵌入式系统里链接脚本至关重要——它决定了代码烧在 Flash 的哪里、栈放在 SRAM 的哪个位置、中断向量表的起始地址等底层细节。

链接脚本由三大核心指令组成：MEMORY 块定义物理内存区域（起始地址、长度、属性 rwx），SECTIONS 块定义各段的放置规则（.text 代码段、.data 已初始化数据、.bss 未初始化数据、.rodata 只读常量），PROVIDE/ENTRY 导出符号给 C 代码使用（如 _estack 栈顶、_sdata 数据段起始）。

启动流程的关键一环：从 Flash 拷贝 .data 段到 RAM 是由 startup 汇编代码完成的，它会读取链接脚本导出的 _sidata（Flash 里 .data 的 LMA 加载地址）、_sdata/_edata（RAM 里的 VMA 运行地址），循环拷贝；随后把 _sbss 到 _ebss 区间清零（C 标准要求 BSS 默认为 0）；最后跳转到 main()。

VMA（Virtual Memory Address 运行地址）与 LMA（Load Memory Address 加载地址）的区别是嵌入式开发者必须理解的概念。.text 段 VMA=LMA（都在 Flash），.data 段 VMA 在 RAM 但 LMA 在 Flash（AT 指令），这就是 “Flash 映像中为什么能看到 .data 的初始值” 的原因。

易混淆点

  1. VMA vs LMA：.data 段运行时在 RAM（VMA），但映像里存在 Flash（LMA），启动代码负责拷贝
  2. KEEP 指令：防止链接器把未被引用的段（如中断向量表）优化掉
  3. 段对齐：. = ALIGN(4) 是必需的，Cortex-M 某些访问要求 4 字节对齐，否则 HardFault

  4. RAM AT>FLASH 语法：左侧是 VMA，右侧是 LMA，两者可不同

面试常见问法

Q1：为什么需要链接脚本？编译器生成的 .o 直接链接不行吗？
A：.o 文件里段的地址是相对的，链接脚本告诉链接器每个段放在物理内存的哪里。嵌入式没有 OS 来加载程序，必须在链接阶段就确定地址。

Q2：.data 段和 .bss 段的区别？
A：.data 存放已初始化且非零的全局/静态变量，在 Flash 里有初始值副本，启动时拷贝到 RAM。.bss 存放未初始化或初始化为 0 的变量，Flash 里不占空间，启动时清零。

Q3：如何把一段代码固定放到 RAM 里执行（如中断处理函数加速）？
A：C 代码用 attribute((section(“.ram_func”))) 标记，链接脚本中新增 .ram_func 段并设置 >RAM AT>FLASH，启动时一并拷贝。
