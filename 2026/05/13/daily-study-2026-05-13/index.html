🔧 嵌入式知识：嵌入式 HTTP 服务、资源限制与安全边界

本节已于 2026-07-14 按官方资料重新校订；旧版自动生成内容已移除。

HTTP 语义不会自动提供安全边界

嵌入式 HTTP 服务首先要限定资源：最大请求行、header 数量/总长度、body 长度、解析深度、连接数、每阶段超时和响应大小。解析必须支持 TCP 分片且拒绝歧义 framing，不能假设一次 recv() 得到完整请求。路径先按协议一致地解析和规范化，再映射到固定资源；禁止目录穿越、整数溢出和由客户端长度直接驱动无界分配。

方法应符合明确语义：读取使用安全方法，改变状态的操作定义幂等键或冲突处理，不能因为客户端超时重试就重复执行“恢复出厂”。每个端点分别做认证和授权，区分查看状态、改配置、触发执行、上传固件和读取敏感日志。仅处于内网、使用非标准端口或隐藏 URL 都不是认证。

HTTPS 需要完整的 TLS 实现、可信根、主机/设备身份、密钥保护、随机数和更新策略；“启用了加密套件”并不代表证书验证正确。受限设备若无法安全终止 TLS，可在受控网关终止外部连接，但网关到设备的物理与网络信任边界必须被明确，不能把明文管理接口暴露给同一广播域所有主机。

工程例子：设备配置 API

GET /v1/config 只返回非秘密字段；PUT /v1/config 要求经过授权的客户端、固定 content type、有限 JSON 大小和 schema 校验。服务端先构建候选配置、做跨字段验证，再以原子存储提交，返回新 ETag/版本。客户端用条件请求避免覆盖他人的更新。重启、固件升级等高风险动作使用单独权限、幂等 operation ID 和明确状态查询，不在响应尚未发出前立即切断连接。

验证清单

  • 对请求行、header、chunk/body、百分号编码、重复字段、异常方法和慢速分片做模糊测试，确认内存、线程和套接字有硬上限。
  • 建立端点—身份—权限矩阵，测试未认证、越权、过期凭据、重复请求、并发修改和恢复出厂后的凭据状态。
  • 用互操作客户端验证状态码、content type、长度、缓存、条件请求和幂等语义；错误响应不能泄露密钥、栈或内部路径。
  • 检查 TLS 版本/套件、证书链、名称或设备身份、随机数、私钥存储、轮换和吊销；测试错误证书与时钟异常。
  • 在最大并发、慢客户端、断线、低内存和网络风暴下做压力测试，确保控制/安全任务不被 HTTP 饿死。
  • 固件上传同时验证传输长度、镜像结构、签名、防回滚和掉电恢复；TLS 只保护传输，不能替代镜像签名。

适用边界与常见误区

RFC 9110 定义 HTTP 语义，不规定你的业务授权和资源预算；TLS 1.3 标准也不能证明具体库配置无缺陷。浏览器访问的管理页还需考虑跨站请求与来源策略，机器 API 则应避免把长期令牌放进 URL。自签名证书并非天然错误，但“接受任意自签名证书”失去身份验证。把解析器放在低优先级任务能限制 CPU 影响，却不能修复越界读取或无界分配。

自测

设备 API 已使用 HTTPS，是否可以省略固件镜像签名？不可以。HTTPS 保护一段连接，签名用于验证落盘镜像的发布者与完整性，并支撑离线启动验证。
