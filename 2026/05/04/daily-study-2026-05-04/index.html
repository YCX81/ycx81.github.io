🔧 嵌入式知识：中断底半部分与任务延迟处理

原理

在 Linux / RTOS / 裸机驱动里，中断处理历来要做两件矛盾的事：响应快（尽早返回让总线恢复）和逻辑全（通常数据还要解析、传输、唤醒等后续工作）。解决办法是分成两半：

  • 顶半部（Top Half / ISR / 硬中断）：关中断运行，做最少最紧急的事——读清中断状态位、把裸数据塞进缓冲、切到下一步。必须短，通常 < 几微秒；期间其他同/低优先级中断被阻塞。
  • 底半部（Bottom Half / 软中断 / 延迟处理）：开中断后在更宽松的上下文里做 parsing、唤醒任务、业务逻辑。不影响实时性。

Linux 底半部的三种经典机制：

  1. softirq / tasklet：kernel 内部软中断，优先级高于进程、运行在中断退出后立即执行。软实时好，但不能睡眠。用于网卡、块设备 RX。
  2. workqueue：把工作排入内核线程运行，可睡眠、可阻塞、可申请大内存，延迟稍大（μs~ms）。用于需要复杂处理或需要锁睡眠的场景。
  3. threaded IRQ：request_threaded_irq() 把整个 handler 放进独立内核线程，既可睡眠又保持 IRQ 框架。现在很多驱动的首选。

RTOS 里同样套路：

  • ISR 只做最简单的 xTaskNotifyFromISR / xQueueSendFromISR，然后 portYIELD_FROM_ISR。
  • 任务在 block 状态等通知，被唤醒后做完整处理。
  • 如果连入队都复杂，可以用 Deferred Interrupt Handler / Daemon Task（FreeRTOS xTimerPendFunctionCallFromISR）把函数交给计时器守护任务去执行。

易混淆点

  1. tasklet ≠ process：tasklet 运行在 softirq 上下文，禁用抢占，不能睡眠；但多个 tasklet 的实例不会并发到同一核。
  2. **中断上下文禁 malloc(GFP_KERNEL)**：kmalloc 在 GFP_KERNEL 可能睡眠；中断内只能 GFP_ATOMIC 或预分配。
  3. 任务优先级倒置：底半任务优先级太低，关键消息被一般业务抢占，真正实时要把底半任务放到较高优先级再让应用层更低。
  4. 中断延迟 vs 中断抖动：嵌入式实时分析要的是抖动（latency 的方差），底半机制有助于降低抖动。
  5. workqueue 不是马上执行：调度粒度取决于当前系统负载，不适合 μs 级延迟敏感任务；要 μs 级用 softirq/tasklet/HRT。
  6. 中断重入：顶半没处理完再来一次中断会导致状态错乱。需要要么关局部中断、要么用 IRQF_ONESHOT / 关个电平触发源。

面试常见问法

Q1：为什么要做底半部？直接在中断里处理不行吗？
A：① 中断里关中断时间长会拉高系统整体中断延迟，掉包、掉帧、丢信号；② 中断上下文受限（不能睡眠、不能拿 Mutex、不能申请大内存）；③ 分工明确便于维护。底半让“重活”有机会在更宽松的上下文里做。

Q2：softirq、tasklet、workqueue 怎么选？
A：对延迟最敏感、不需要睡眠的高频场景——softirq/tasklet。需要睡眠、可能长耗时、对延迟不敏感的——workqueue。既要睡眠又要 IRQ 上下文语义的，现代驱动用 threaded IRQ 最省心。

Q3：FreeRTOS 里用 Deferred Interrupt Handler 的收益？
A：把 ISR 里的函数调用“押”到一个守护任务执行，既保留从 ISR 产生事件的能力，又让真正业务逻辑有任务上下文，可以调用普通 API、记录日志、使用互斥量。

Q4：portYIELD_FROM_ISR 什么时候必须调用？
A：当某个 FromISR API 返回 pdTRUE 表示有更高优先级任务被唤醒时，必须 portYIELD_FROM_ISR 让调度器立刻切换，否则要等到下一次中断/tick。漏调会导致高优先级任务被“拖着”。
