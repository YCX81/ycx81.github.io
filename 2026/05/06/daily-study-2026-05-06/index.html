🔧 嵌入式知识：代码静态分析

原理

代码静态分析（Static Code Analysis）在不执行程序的情况下，通过扫描源代码或字节码发现潜在缺陷、安全漏洞、规范违反。它和动态测试（单元测试、fuzzing、valgrind）是互补关系：静态能覆盖“罕见路径”，动态能发现“运行时 heisenbug”。

技术栈由浅到深：

  1. 词法/语法检查：命名风格、排版、缩进、行长。代表：clang-format、Prettier、flake8。
  2. 规则/模式匹配：基于 AST 的规则引擎，查“XXX 必须 YYY”。代表：ESLint、Semgrep、cppcheck。
  3. 数据流分析：沿控制流传播变量抽象值，找空解引用、UAF、数组越界、整溢、未释放。代表：Clang Static Analyzer、Infer。
  4. 符号执行 / 抽象解释：路径敏感地枚举条件约束，发现深层 bug。代表：KLEE、Coverity、CodeSonar。
  5. 污点分析（Taint Analysis）：跟踪“不受信任输入”流向，找 XSS/SQLi/命令注入。代表：CodeQL、Fortify。
  6. 合规扫描：按 MISRA-C、CERT C、AUTOSAR C++14 等强规则集检查。代表：PC-lint、Helix QAC。

工程流程：

  • 本地开发：IDE 插件实时提示，提交前可自动 lint --fix。
  • Pre-commit：hook 跑 clang-format + clang-tidy 子集。
  • Pull Request：CI 跑完整扫描，diff-gate（新增告警 = 阻塞合并）。
  • 夜间全量扫描：耗时的深度分析，结果进 SonarQube 仪表盘跟踪趋势。
  • 发布前：合规扫描（MISRA / CERT），生成审计报告。

关键代码/配置

易混淆点

  1. 假阳性：越精确的分析器越容易假阳性；抑制要带理由，不能无脑关。用 // NOLINTNEXTLINE(reason) 或 /* cppcheck-suppress ... */。
  2. 分析器要能看到完整上下文：头文件路径、宏、目标编译器配置必须给全，否则覆盖率很差。CMake 的 compile_commands.json 是黄金标准。
  3. LOC-Based vs Diff-Based gate：新项目全量达标不可能，实用做法是“基线冻结 + 新增必须 0 告警”。
  4. 静态 ≠ 测试：静态分析再强也不能替代单元测试；两者必须并用，尤其是业务逻辑正确性问题静态是抓不到的。
  5. 安全扫描 vs Bug 扫描：Fortify / CodeQL / Semgrep 偏安全漏洞；Coverity / Clang Analyzer 偏功能性缺陷，关注点不同。

面试常见问法

Q1：为什么编译器警告（-Wall）不够用？
A：编译器只做单编译单元的浅层检查，过程间分析、条件约束、污点追踪都做不了。例如“if(ptr) ... ptr->x”这种“先判空再解引用”的问题，编译器不会报，静态分析器会。另外编译器不懂 MISRA、CERT 等规范。

Q2：MISRA-C 合规怎么做？
A：① 选工具（PC-lint Plus / Helix QAC / Parasoft C/C++test）；② 按项目 ASIL/DAL 确定子集和例外清单；③ 扫描 + Deviation 管理——每条豁免都要写理由并审批；④ 定期审计 Deviation 合理性；⑤ 最终生成合规证据报告。

Q3：SonarQube 扮演什么角色？
A：它是汇总+可视化平台，不是扫描器，接收 clang-tidy / cppcheck / Coverity / CodeQL 结果，做去重、分类、历史趋势、质量门禁（Quality Gate）。开发者 PR 里看得到“本次引入的告警”，管理层看得到项目总体趋势。

Q4：如何说服团队接受静态分析？
A：① 先从少量规则起步，不堆成”几万告警”的心理冲击；② 把告警挂到真实的线上事故做归因——比如“这条规则本能抓住上次那个 CVE”；③ 渐进式基线：每迭代消 5%；④ 让规则真正帮开发者（IDE 实时提示、自动修复），而不是只在 CI 卡人。
