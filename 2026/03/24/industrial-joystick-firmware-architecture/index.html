工业摇杆产品线软件架构设计（v5 完整版）

发表于 2026-03-24 分类于 嵌入式架构 本文字数： 19k 阅读时长 ≈ 1:08

工业摇杆产品线固件架构设计完整方案：从问题分析到 v5 演进，覆盖驱动接口层（ISP 拆分）、三层输入模型、显式信号管线、Output Adapter、Profile 驱动协议层、E-Gas 三层功能安全架构（SIL 2 / PL d）、RTOS 任务架构和 CMake 构建系统。

一、问题分析

一个工业摇杆产品线通常面临三个维度的变化：

如果用排列组合，4×4×3 = 48 种配置。传统做法是每种配置一个工程，维护噩梦。
正确的做法是用组合而非继承，通过架构设计让这三个维度正交独立。

3.2 CAN 端口接口（v5 改进：callback 内收到平台层）

v4 原设计通过 register_rx_cb(can_rx_callback_t cb) 将 ISR 回调暴露给上层。这导致：

  • 上层必须在回调中处理 ISR 语义（重入约束、栈大小限制）
  • STM32 弱符号回调、NXP 中断分发句柄、PC 线程的差异泄漏到应用层
  • 不同平台的回调注册方式完全不同

v5 改为 端口模型：callback 只存在于平台适配层内部，上层通过 try_recv / wait_recv 消费帧。

3.4 平台注册机制（DIP 改进：依赖方向完全反转）

原设计中 app_config.c 通过 #if defined(PLATFORM_STM32G4) 直接引用平台符号，编译时依赖方向从上往下。改进后，平台层主动向上注册，上层代码完全不知道具体平台的存在。

为什么不用 __attribute__((constructor))？

对比项 __attribute__((constructor)) 显式 platform_init()
MISRA C 合规 违反 Rule 1.2（实现定义行为） 合规
初始化顺序 不确定，跨编译单元无法保证 代码中显式控制
可调试性 在 main() 之前执行，断点难设 普通函数调用，随时断点
功能安全认证 IEC 61508 / ISO 13849 审核不通过 可审核、可追溯
可移植性 GCC 专属，IAR/ARMCC/Green Hills 不支持 纯 C 标准，所有编译器通用

链接器决定哪个平台的 platform_init.c 参与编译，上层代码零修改。每个平台只需提供自己的 platform_init() 实现即可。

3.7 异步原语层（v5 新增：ISR 边界解耦）

3.7.1 设计动机

架构中存在三类异步交互模式，它们的平台差异大于 RTOS 差异：

场景 ISR 侧行为 任务侧行为 平台差异
ADC DMA 完成 更新双缓冲索引 + 置位通知 取最新完成帧 STM32 HAL callback vs NXP eDMA callback vs PC 定时器
CAN RX 帧入环形缓冲 + 唤醒消费者 阻塞/轮询取帧 STM32 弱符号回调 vs NXP 中断分发句柄 vs PC 线程
事件总线 ISR 发布 事件入队列 延迟分发 CMSIS-RTOS 依赖 ARM IPSR 寄存器自动检测，非 ARM 平台不可用

CMSIS-RTOS v2 是 OS 抽象，不是 I/O 语义抽象。 它解决了任务调度/互斥/延时的可移植性，但不解决 ISR→任务数据传递的语义统一。需要一层极薄的异步原语，专门覆盖 ISR→任务边界。

3.7.2 职责边界（硬约束）

规则：

  • 任务调度、任务间互斥、定时器 → 继续用 CMSIS-RTOS v2
  • ISR 投递数据/唤醒消费者 → 使用 async_notify_t / async_queue_t
  • 内存可见性保证 → 使用 async_write_barrier() / async_read_barrier()

volatile 只约束编译器重排序，不足以表达跨核/总线/外设 DMA 的可见性。双缓冲模式中必须显式插入屏障。

四、输入采集层（v5 改进：三层输入模型 + 异步解耦）

4.1 设计动机

原设计假设”轴 = ADC 通道”，joystick_update() 直接调用 adc->read(ch, &raw)。但实际工业摇杆的输入源远比这复杂：

关键约束：

  1. 轴和按键共用 MUX+ADC 资源——不能各自独立读取，必须协调扫描顺序
  2. 按键采用逐通道 5V→2.5V 分压——每个按键占独立 MUX 通道，按下时读 ≥2.5V，支持多键同按
  3. 不同产品硬件差异大——有的用 MUX，有的直接 ADC 多通道，有的用 DMA 连续采集
  4. 拉模式与推模式混合——MUX/直接 ADC 是同步拉取，DMA/SPI DMA 是异步推送

若将 MUX+ADC 作为整体资源封装进输入源，会导致轴源和按键源耦合在同一个扫描器上，缺乏平台化能力。而 v4 的 board_scanner_t.scan() 同步模型让 DMA 冒充同步扫描，掩盖了异步语义差异。因此 v5 将输入侧正式拆为三层。

4.3 input_acq_t — 统一采集接口（v5 新增）

替代 v4 的 board_scanner_t。不再区分”扫描”和”取帧”，统一为单次原子获取：

seq 的三个用途：

  1. 判断是否有更新：curr_seq == last_seq → 无新数据，沿用上次
  2. 检测丢帧：curr_seq - last_seq > 1 → 采集速率快于消费速率

4.4 scan_buffer_t — 通用样本载体

scan_buffer_t 是整个输入采集层的解耦核心：

  • 上游（input_acq_t）只写入它
  • 下游（输入源）只读取它
  • 它本身不依赖任何硬件接口，可在 PC Mock 上直接填充测试数据
  • v5 明确：它是快照对象，不是 DMA 工作缓冲

4.6 Sensor Backend — 异步推送实现（DMA / SPI DMA）

DMA 不是扫描策略，而是采集机制。v5 使用双缓冲 + 序号 + 内存屏障，上层永远不接触 ISR callback。

关键设计：DMA ISR 只做三件事：① 记录完成的缓冲区索引 ② 更新序号 ③ 发通知。不做任何数据拷贝或业务逻辑。上层通过 acquire_latest() 从非活跃缓冲区取稳定快照。

五、信号处理管线架构（v4 改进：统一为显式 pipeline，删除泛型 chain）

5.1 内部信号表示：无符号原始范围，不归一化

信号处理管线不做归一化，全程保持传感器原始范围。值域转换由独立的 output_adapter 层在协议打包前一次性完成（见第六章）。

传感器 原始范围 中心值 数据类型
12-bit ADC 0–4095 ~2048（校准值） uint16_t
14-bit Hall 0–16383 ~8192（校准值） uint16_t

为什么不在管线内归一化：

  • 精度损失是真实的 — 14-bit 传感器抖动 ±2~3 LSB，一次整数除法映射可能把信号淹没在量化噪声中
  • 校准不对称时引入非线性 — 正/负向量程不同，归一化会在零点附近产生增益跳变
  • 管线各阶段不需要统一量纲 — 故障检测在 uint16_t 域，滤波/死区在 int32_t 域围绕零点操作，天然不关心绝对范围

5.2 设计动机

原设计中 joystick_core.c 的 process_axis() 函数同时负责 ADC 采集、故障检测、校准计算、死区处理、按键消抖五个职责。任何一个需求变化（换消抖算法、换死区曲线、加数字滤波）都要修改这个文件。

改进：拆分为信号处理管线（pipeline），每个阶段是独立模块，各自只有一个变更理由。

v4 变更： v3 同时存在泛型 signal_chain_t（int16_t 统一签名）和显式 axis_pipeline_t（uint16_t→int32_t 分界）两套接口。v4 删除泛型 chain，只保留显式 pipeline。理由：管线阶段顺序固定，真正变化的是参数和策略，泛型 chain 的灵活性不被需要，反而引入类型不匹配和概念混淆。

11.7 事件总线架构（v5 改进：控制面/数据面分离 + 队列后端注入）

11.7.1 控制面与数据面分离原则

v4 的 event_bus 同时承载状态事件和高频数据，v5 明确分离：

面 内容 传输机制 频率
控制面 故障事件、状态切换、模式变更、CAN Bus-Off event_bus（观察者模式） 低频（事件驱动）
数据面 ADC 样本、CAN RX 帧、SPI 传感器值 input_acq_t / can_port_t（快照+队列） 高频（周期性）

硬约束：event_bus 不承载高频原始数据。ADC/CAN/SPI 的周期性样本走 input_acq_t.acquire_latest() 或 can_port_t.wait_recv()。

11.7.2 ISR 安全：队列后端注入

v4 的 event_bus_publish_from_isr() 依赖 CMSIS-RTOS v2 的 IPSR 寄存器自动检测（ARM Cortex-M 专属）。v5 将队列后端改为可注入策略，支持跨平台：

各平台实现：

安全约束不变：L2 安全通道不依赖 event_bus，最多读共享快照或独立采样。L3 硬件关断路径（GPIO 直驱安全继电器）完全不经 event_bus/RTOS。

11.8 时序要求与优先级反转防范

关键设计规则：

  • safety_task 中禁止使用 mutex（避免优先级反转导致安全超时）
  • 安全任务只读取数据，使用 atomic 或 volatile 访问
  • protocol_status 改为 atomic 单写者（proto_task 写，safety_task 无锁读）
  • CAN RX 中断处理时间 < 5μs（只做 osMessageQueuePut 入队列）
  • 所有 mutex 创建时指定 osMutexPrioInherit（优先级继承）
  • main_task 的 mutex 获取设置 5ms 超时（避免死锁）
  • 所有周期任务使用 osDelayUntil() 绝对延时（避免周期漂移）

十二、功能安全架构（v4 新增：E-Gas 三层模型，SIL 2 / PL d）

12.1 安全需求与标准对照

工业摇杆用于工程机械、农业设备等安全关键场景，需满足：

标准 等级 核心要求
IEC 61508 SIL 2 DC ≥ 90%，安全功能与控制功能独立，程序流监控
ISO 13849 PL d, Cat.3 双通道结构，单一故障不丧失安全功能

12.2 为什么 v3 不满足

v3 的 safety_task 是”旁观者”——高优先级执行，但与 main_task 共享 RTOS、内存空间、ADC 驱动、事件总线。优先级隔离 ≠ 独立性。RTOS 内核 bug、堆栈溢出、系统节拍丢失都是共因失效点。
