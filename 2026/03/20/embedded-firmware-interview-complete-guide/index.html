2.5 CCM RAM 深度解析

CCM = Core Coupled Memory，直接连在 CPU 的 D-Bus 上。

CCM vs SRAM 性能对比：

特性 SRAM CCM RAM
总线连接 AHB 总线矩阵 CPU D-Bus 直连
CPU 访问速度 零等待（无竞争时） 零等待（永远）
DMA 竞争 有，DMA 和 CPU 抢总线 无，DMA 访问不了
DMA 可访问 可以 不可以
可执行代码 可以 不可以（F4）

实测数据（STM32F407，168MHz，FOC 20kHz 中断，同时 ADC+SPI DMA）：

为什么 CCM 存线程栈（三大理由）：

  1. 性能确定性 — CPU 通过专用 D-Bus 访问 CCM，不受 DMA 竞争影响，FOC jitter 从 1.8μs 降到 0.3μs
  2. 硬件隔离 — DMA 不可访问 CCM，从硬件层面消除”DMA 配置错误踩栈”故障（FMEA 中可直接排除）
  3. MPU 精细保护 — 安全监控线程栈设为仅特权模式可访问，应用线程溢出也踩不到

4.5 USB CDC 在 ISR 中发送的问题

结论：ISR 里不能调 CDC_Transmit_FS()，否则大概率卡死。

原因：

  1. CDC_Transmit 非原子操作，主循环和 ISR 同时调用会竞争覆盖缓冲区指针
  2. USB 中断回调负责清 TxState，若 ISR 优先级 ≥ USB 中断 → TxState 永远不清零 → 永远返回 BUSY

正确做法（ISR 写缓冲区，主循环发送）：

USB FS 吞吐量上限： 64 bytes/ms = 64 KB/s。FOC 20kHz 每次发 8 字节 = 160 KB/s → 超了，必须降采样（每 4 次发一次）。

5.2 FOC 前馈补偿大全

前馈的本质：不等误差出现，把已知的扰动提前算出来补上。

1. 交叉耦合 + 反电动势前馈（最核心）：

$$V_q = R_s i_q + L_q \frac{di_q}{dt} + \underbrace{\omega_e L_d i_d}{\text{交叉耦合}} + \underbrace{\omega_e \psi_f}{\text{反电动势}}$$

没有前馈：高速时 PI 输出全用来抵消反电动势，没余量控制电流。
有前馈：PI 只处理模型误差，输出小、响应快、不饱和。

5.5 电流环为什么优先于电压模式

电压模式 电流环
门槛 低（无需采样电阻+运放） 高（需同步 ADC+运放）

双芯片架构（nRF52840 + STM32）的理由：

BLE 射频时序和 FOC 中断时序（20kHz）互相干扰。分两颗芯片，nRF 专心跑 BLE，STM32 专心跑 FOC，通过 SPI 解耦。SPI 1MHz，每次 16 字节，延迟 < 20μs，不影响控制实时性。

BLE 高频追问：

  • Q: 数据可靠性怎么保证？ Notify 无 ACK 可能丢包，关键指令用 Write with Response（有 ACK）+ 应用层序列号检测 + 重要参数写后 Read-back 验证。
  • Q: MTU 是什么？ 默认 23 字节（有效 20），可协商到 247 字节，减少分包次数提高吞吐量。
  • Q: OTA 怎么做？ MCUboot + SMP，手机 BLE 传固件分包，写备用分区，校验后重启切换。

8.3 综合面试模拟题速查

Q: 你怎么设计一套支持多款硬件的通用固件？

分三层：业务层（算法+状态机）、HAL 抽象层（接口定义）、平台实现层（具体硬件驱动）。业务层通过函数指针接口调用硬件，不直接操作寄存器。初始化时通过依赖注入把具体平台实现传入业务模块。换硬件只需新增一套平台实现，业务代码不动。Mock 平台可以在 PC 上跑单元测试，CI 自动验证算法正确性。

Q: 如何保证固件质量？

三道防线：①Mock 平台单元测试——核心算法在 PC 上跑，每次 push 自动执行，FOC 测试里有简单电机模型验证电流跟踪精度；②CI 交叉编译 + 静态分析——ARM-GCC 确保能烧进去，cppcheck 抓未初始化变量，检查固件大小不超限；③PR 必须单元测试全过 + Code Review 才能合并 main。

Q: PendSV 是什么，为什么优先级要最低？

PendSV 是 Cortex-M 专为 RTOS 上下文切换设计的异常。优先级最低确保所有外设中断处理完后才做切换，防止在中断嵌套中切换线程导致栈混乱，也防止打断 FOC 等高实时性中断。

Q: FreeRTOS Tickless 是怎么工作的？

所有任务阻塞时，内核从延时列表读出最早唤醒时刻，计算可以睡多久。Port 层把 tick 数换算成 LPTIM 计数值（LSE 32.768kHz，Stop 模式不停），设好 ARR 进 Stop 模式。唤醒后判断是时间到了还是外部中断提前唤醒，调 vTaskStepTick 补偿 tick 数。整个过程对上层任务完全透明。

Q: CCM RAM 存线程栈有什么好处？

三点：①CPU 专用 D-Bus 访问 CCM 零等待，不受 DMA 竞争，FOC 中断 jitter 从 1.8μs 降到 0.3μs；②DMA 不可访问 CCM，硬件层面消除”DMA 踩栈”故障；③MPU 可把安全线程栈设为特权模式，应用层改不了。
