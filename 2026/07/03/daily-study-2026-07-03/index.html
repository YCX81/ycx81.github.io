🔧 嵌入式知识：DMA 与 D-Cache 一致性

本节已于 2026-07-14 按官方资料重新校订；旧版自动生成内容已移除。

两个主设备，三份可能不同的视图

带 D-Cache 的 MCU 上，CPU 可能只修改了 cache line，外部存储中的内容尚未更新；DMA 则通常直接读写内存，不经过 CPU 的 D-Cache。于是会出现两类典型错误：DMA 发送前没 clean，DMA 读到旧数据；DMA 接收后 CPU 没 invalidate，CPU 继续读到缓存中的旧数据。

对齐不只是性能问题

缓存维护按 cache line 工作。若缓冲区与别的数据共用一条 line，invalidate 可能丢掉邻近变量尚未写回的修改，clean 也可能影响超出缓冲区的区域。稳妥做法是让 DMA 缓冲区按 line 大小对齐并独占完整 line，长度向上取整；line 大小、API 的地址类型和前后屏障要求都以具体内核和 SDK 为准。

另一种方案是把 DMA 缓冲区放在不可缓存或硬件一致的内存区，但代价和可用性取决于 MPU、总线矩阵及芯片实现。还要确认 DMA 能否访问该内存：某些紧耦合内存只对 CPU 可见。

自测

DMA RX 完成中断到了，是否就能直接解析缓冲区？在非一致 D-Cache 系统上不能。先确认 DMA 完成和内存可见性，再按平台规则 invalidate；而且不能让维护操作误伤共享 cache line。
