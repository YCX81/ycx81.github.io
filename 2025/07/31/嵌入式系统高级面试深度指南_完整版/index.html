嵌入式系统高级面试深度指南

第一章：C/C++ 编程基础与编程实操

1.1 volatile 关键字在嵌入式开发中的典型使用场景

核心概念

volatile 告诉编译器：该变量的值可能在程序控制流之外被改变，编译器不得对其进行优化（如缓存到寄存器、消除”冗余”读取、重排序等）。

典型使用场景

场景一：硬件寄存器访问

如果不加 volatile，编译器可能认为第一次赋值无意义（被第二次覆盖），直接优化掉。

场景二：中断服务程序（ISR）与主循环共享变量

不加 volatile，编译器可能将 g_data_ready 缓存到寄存器中，主循环永远读到初始值 0，导致死循环。

场景三：多线程/多任务共享变量（RTOS 环境）

场景四：内存映射 I/O 状态轮询

场景五：DMA 缓冲区

不加 volatile 的典型后果

问题 表现
中断标志轮询失效 主循环死等，永远检测不到标志变化
寄存器写入被优化 连续写入被合并或消除，外设行为异常
状态读取过时 使用了寄存器缓存的旧值，逻辑判断错误
Debug 正常 Release 异常 Debug 模式关闭优化正常，开优化后出 Bug

深度拓展：volatile 不能解决的问题

  • 不保证原子性：对 32 位 MCU，64 位变量的读写仍非原子操作
  • 不保证内存屏障：在多核系统中不能替代内存屏障指令
  • 不保证线程安全：共享变量还需配合临界区或互斥锁

1.2 const 与 #define 定义常量的本质区别

核心对比

特性 const #define
处理阶段 编译期 预处理期（文本替换）
类型检查 有，编译器进行类型检查 无，纯文本替换
作用域 遵循 C 的作用域规则 从定义处到文件末尾（或 #undef）
调试可见性 符号表中有名字，可调试 无符号名，调试器中显示为数字
内存占用 可能占用 ROM（Flash） 每处使用都内联展开，可能增大代码体积
可取地址 可以 &my_const 不可以

最佳实践总结：

  • 数值常量优先用 const（或 enum）
  • 寄存器地址/位掩码用 #define（行业惯例）
  • 条件编译标记必须用 #define
  • 查找表、校准数据用 const 数组（存 Flash 省 RAM）

深度拓展：static const vs #define 在嵌入式中的 Flash 占用

1.3 static 关键字的三种含义

场景一：函数内部的 static 局部变量

含义：变量生命周期延长到整个程序运行期间，但作用域仍限于函数内部。

存储位置：放在 .data（已初始化）或 .bss（零初始化）段，而非栈上。

场景二：文件作用域的 static（全局变量/函数）

含义：将符号的链接属性限制为内部链接，即只在当前编译单元（.c 文件）可见。

嵌入式最佳实践：

  • 模块内部函数和变量都应加 static，实现信息隐藏
  • 相当于 C 语言版的”私有成员”
  • 减少全局命名空间污染，避免不同模块间的命名冲突

场景三：C++ 类的 static 成员

2.3 NVIC 中断控制器与优先级分组

NVIC 工作原理

NVIC（Nested Vectored Interrupt Controller）是 Cortex-M 内核集成的中断控制器：

  • 支持嵌套：高优先级中断可打断低优先级中断
  • 尾链优化：连续中断无需完整的入栈/出栈过程
  • 自动入栈：硬件自动保存 R0-R3, R12, LR, PC, xPSR（8个寄存器）
  • 晚到优化：如果入栈过程中有更高优先级中断到达，直接处理高优先级中断

中断优先级分组

Cortex-M 使用 8 位优先级字段（实际实现 3-8 位），分为两部分：

2.6 Thumb-2 指令集

概念

  • Thumb：16 位指令，代码密度高，但功能受限
  • ARM：32 位指令，功能完整，但代码体积大
  • Thumb-2：混合 16 位和 32 位指令，兼顾密度和功能

为什么 Cortex-M 只支持 Thumb 模式

  1. 代码密度：嵌入式 Flash 空间宝贵，16 位指令可减小代码体积 30%+
  2. 简化设计：不需要 ARM/Thumb 模式切换逻辑，降低功耗和面积
  3. Thumb-2 已足够：32 位 Thumb-2 指令覆盖了绝大多数 ARM 指令功能
  4. 性能无损：Thumb-2 的性能接近纯 ARM 模式，但体积小得多
